# Security Priority Fixes

This table is the live working queue for findings from the v1.0 quality-gate security audit. Keep it updated as each item moves from identified to fixed to staging-verified.

Status values:

- `identified`: confirmed or plausible risk documented, not fixed yet.
- `patched-local`: code/config patch exists on this branch, not deployed to staging yet.
- `verified-staging`: fix has been deployed and verified against staging.
- `accepted`: intentionally deferred with rationale.

| ID | Priority | Severity | Area | Finding | Evidence | Attack path | Fix plan | Status | Verification needed |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |
| SEC-001 | P0 | High | Backend cache / availability | Unbounded in-memory response cache can be abused for memory DoS. | `api/internal/handlers/middleware.go` `ResponseCache.entries`, keyed by `r.URL.RequestURI()`. | Anonymous or distributed API clients request many unique filter/query combinations; process memory grows until degraded/restarted. | Add max entries plus LRU/TTL cleanup; cap query string length; normalize cache keys; avoid caching high-cardinality or empty filtered responses. | patched-local | Add unit test for eviction; run burst/high-cardinality request check; confirm memory stays bounded. |
| SEC-002 | P0 | High/Medium | Backend rate limiting / proxy trust | Per-IP rate limit trusts `X-Forwarded-For` without proving the sender is the trusted proxy. | `api/internal/handlers/middleware.go` `clientIP`; `deploy/Caddyfile.example` currently overwrites headers. | Future proxy misconfig or direct exposure lets attacker spoof many client IPs and bypass rate limits. | Trust forwarded headers only from known proxy IPs, or have Caddy pass a sanitized internal client-IP header and ignore user-supplied XFF. | patched-local | Unit test spoofed XFF; staging test confirms user-supplied XFF cannot change bucket identity. |
| SEC-003 | P0 | Medium | Observability / information disclosure | Public `/health` exposes raw ingestion errors and operational details. | `api/internal/handlers/health.go` exposes `LastIngestion.Error`; staging previously exposed connection failure details. | Anonymous user polls `/health` to learn internal IPs, provider errors, run cadence, version, and outage timing. | Split public liveness from private diagnostics; redact `error` publicly; keep detailed error only in logs/admin diagnostics. | patched-local | Staging `/health` shows status/version/minimal freshness only; private diagnostics remain available to operators. |
| SEC-004 | P1 | Medium | Monitoring logic | `/health` always returns HTTP 200 even when degraded. | `api/internal/handlers/health.go` always writes `http.StatusOK`. | Monitors or load balancers that only check HTTP status miss ingestion failure; degraded data can look healthy. | Add separate `/live` and `/ready`; return non-2xx from readiness when ingestion is stale/degraded if deployment policy requires fail-closed. | patched-local | CI/API test for degraded readiness status; staging monitor uses readiness endpoint. |
| SEC-005 | P1 | Medium | Frontend security headers | Staging frontend lacks CSP, nosniff, frame, referrer, and permissions headers until current patch is deployed. | Live staging header check showed missing headers; `web/vercel.json` now contains local header patch. | XSS or clickjacking impact is higher without browser-enforced guardrails. | Deploy current `web/vercel.json`; tighten CSP host allowlist after browser verification. | verified-staging | Head request to staging confirms CSP, `X-Content-Type-Options`, `X-Frame-Options`, `Referrer-Policy`, `Permissions-Policy`. |
| SEC-006 | P1 | Medium | Backend CORS | Staging API currently returns 204 preflight for disallowed origins until current patch is deployed. | Live staging OPTIONS from `https://evil.example` returned 204; local patch checks origin. | Misleading CORS behavior and future credentialed APIs could become cross-origin exposure. | Deploy current strict CORS patch; add tests for allowed, disallowed, and no-Origin requests. | verified-staging | OPTIONS from allowed origin succeeds; disallowed origin returns 403/no ACAO; no-Origin server-to-server requests still work. |
| SEC-007 | P1 | Medium | Input validation | Staging accepts non-integer pagination such as `limit=abc` and falls back to default until current patch is deployed. | Live staging `GET /v1/events?limit=abc` returned 200; local patch returns 400. | Attackers can hide malformed traffic in normal-looking responses and increase cache cardinality. | Deploy current integer validation patch; add handler tests for invalid limit/offset. | verified-staging | Staging returns 400 for `limit=abc` and `offset=abc`; valid bounds still work. |
| SEC-008 | P1 | Medium | Third-party data integrity | Upstream `source_url` is trusted and rendered as an external link. | `web/src/pages/EventDetail.tsx` renders `event.source_url`; normalizer stores first upstream source URL. | Compromised/upstream-bad event data could create trusted-looking phishing links in event details. | Validate scheme and allowed hostnames during normalization or before rendering; show blocked/unverified source state. | patched-local | Unit tests reject `javascript:`, `data:`, and non-allowlisted domains; UI does not render blocked links. |
| SEC-009 | P2 | Low | SQL/input handling | SQL construction is parameterized today but fragile due dynamic string assembly. | `api/internal/database/queries.go` builds WHERE fragments with `fmt.Sprintf`, user values are placeholders. | Future dynamic sort/filter could accidentally inject raw user-controlled SQL. | Add regression tests for malicious filter strings; centralize allowed query fragments; avoid adding raw dynamic order clauses. | patched-local | Tests cover SQLi-looking `country/state`; all values remain parameterized. |
| SEC-010 | P2 | Low | Frontend DOM handling | `innerHTML` is used for marker SVGs, currently with static-only markup. | `web/src/components/Map.tsx` `button.innerHTML`; event titles use attributes/textContent, not HTML. | Future interpolation of event fields into this template would create DOM XSS. | Replace with DOM construction or keep helper private with tests that event data never enters `innerHTML`. | patched-local | Test malicious event title/category does not create executable markup or attributes. |
| SEC-011 | P2 | Low | Repository hygiene | Tracked `.claude/settings.local.json` can accumulate local commands and DSN-style strings. | `.claude/settings.local.json` is tracked and currently modified locally. | Local tool state may eventually capture real credentials or sensitive operator commands. | Remove from tracked source or sanitize; add `.claude/settings.local.json` to `.gitignore` if team agrees. | patched-local | `git ls-files .claude/settings.local.json` returns empty after cleanup; no secrets in tracked local settings. |
| SEC-012 | P2 | Low | Docs/API exposure | `/docs` loads Scalar from CDN and public docs expose endpoint map. | `api/internal/handlers/docs.go` loads `https://cdn.jsdelivr.net/npm/@scalar/api-reference`; `/openapi.yaml` is public. | CDN compromise affects docs page; public docs aid recon. Low risk for public API, higher if admin endpoints are later added. | Self-host docs asset or restrict docs to non-production/private network before adding sensitive endpoints. | patched-local | Production docs policy documented; if restricted, staging/prod return expected status. |
| SEC-013 | P1 | Medium | Container hardening | API container runs as root by default. | `api/Dockerfile` final stage has no `USER`; `docker-compose.*.yml` does not override user. | If the API process or a dependency is compromised, attacker code has root inside the container, increasing impact against mounted files and container runtime weaknesses. | Add non-root app user/group in final image, chown `/app`, run `USER app`; add `read_only`, `cap_drop: [ALL]`, and `no-new-privileges` where compatible. | patched-local | Container starts as non-root; healthcheck passes; write attempts outside expected temp paths fail. |
| SEC-014 | P1 | Medium | Backend rate limiting / availability | Per-IP rate limiter stores token buckets forever and `/health`/docs are not rate-limited. | `api/internal/handlers/middleware.go` `ipRateLimiter.buckets` has no eviction; `api/cmd/server/main.go` applies rate limit only under `/v1/`. | Distributed clients or spoofed trusted headers can grow bucket state; unauthenticated users can hammer health/docs endpoints outside the limiter. | Add TTL eviction/max buckets; consider lightweight global limiter for `/health`, `/docs`, and `/openapi.yaml`; keep uptime probe allowlisting if needed. | patched-local | Unit test bucket eviction; burst test confirms `/v1` and non-`/v1` endpoints enforce expected limits without blocking monitors. |
| SEC-015 | P1 | Medium | Backend scheduler / business logic | Ingestion scheduler has no distributed single-runner guard. | `api/internal/ingestor/scheduler.go` starts scheduler inside every API process and immediately runs ingestion on startup. | If staging/prod is scaled to more than one API replica, each instance ingests on startup/tick, causing duplicate EONET traffic, duplicate ingestion runs, duplicate alerts, and noisy health state. | Add DB advisory lock or split scheduler into a single worker process; document one-replica invariant until fixed. | patched-local | Start two local API instances against same DB and confirm only one obtains scheduler lock/runs ingestion. |
| SEC-016 | P1 | Medium | Supply chain / container images | Runtime and service images use mutable tags instead of digests. | `api/Dockerfile` uses `golang:1.26-alpine` and `alpine:latest`; compose uses `postgis/postgis:15-3.4`, `postgis/postgis:15-3.3`, and unversioned `maxmindinc/geoipupdate`. | A registry-side compromise or upstream tag change can alter build/runtime contents without a code change. | Pin images to immutable digests; schedule controlled digest refreshes; add image vulnerability scan before deployment. | patched-local | CI fails on unpinned images; container scan report attached to release/deploy run. |
| SEC-017 | P1 | Medium | Supply chain / CI | GitHub Actions and OpenSpec installs are tag/latest based rather than immutable. | Workflows use `actions/checkout@v4`, `actions/setup-node@v4`, `actions/setup-go@v5`; `openspec-verify.yml` installs `@fission-ai/openspec@latest`; root `package.json` declares `@fission-ai/openspec: latest`. | Compromised action tag or changed npm latest can alter CI behavior and deployment gates. | Pin GitHub Actions to commit SHAs; pin OpenSpec to an exact version in `package.json` and use `npm ci`/local binary in workflows. | patched-local | `npm ls --depth=0` shows installed exact OpenSpec; workflows use SHA-pinned actions and no global `@latest` install. |
| SEC-018 | P2 | Low/Medium | Supply chain / Docker build reproducibility | Docker build runs `go mod tidy` during image creation. | `api/Dockerfile` line runs `go mod tidy` after copying source. | Build can mutate dependency graph at image build time, masking dirty module state and making local/CI/deploy builds less reproducible. | Remove `go mod tidy` from Dockerfile; enforce `go mod tidy`/`git diff --exit-code go.mod go.sum` in CI. | patched-local | Docker build succeeds without `go mod tidy`; CI fails if module files are not tidy before build. |
| SEC-019 | P1 | Medium | Dependency scanning / CI gate | Vulnerability scans are manual, not deployment gates. | CI runs tests/build but not `npm audit`, `govulncheck`, or container image scanning; root `npm audit` currently fails locally with `ECONNRESET`. | Known vulnerable dependency or base image can ship if tests pass. | Add web/root `npm audit --audit-level=moderate`, `govulncheck ./...`, and container scanner job; handle root OpenSpec dependency so root audit is reliable. | patched-local | CI publishes passing audit outputs; failed audit blocks PR/deploy unless explicitly waived. |
| SEC-020 | P2 | Low | Local compose exposure | Local/demo compose exposes Postgres on all interfaces with default credentials. | `docker-compose.yml` maps `5432:5432` with `vigilafrica/vigilafrica`; `docker-compose.demo.yml` maps `5433:5432` with same defaults. | On shared/dev machines, another host on the network may reach the database if firewall permits. | Bind local DB ports to `127.0.0.1`; keep defaults clearly local-only; consider `.env` overrides for credentials. | patched-local | `docker compose config` shows `127.0.0.1:5432:5432` and `127.0.0.1:5433:5432`. |
| SEC-021 | P1 | Medium/High | Data integrity / state desynchronization | Event upsert can serve stale geometry, category, source URL, and event date after an upstream event changes. | `api/internal/database/repository.go` `ON CONFLICT (source_id)` updates only `status`, `title`, and `raw_payload`; `normalizer.Normalize` can produce newer geometry/source/category fields that are ignored on conflict. | Compromised or simply corrected upstream EONET data reuses a `source_id` with moved coordinates or changed category; API/UI show new title/status with old map position and stale location context, potentially misdirecting responders. This chains with response caching and health freshness because the system still looks "successfully ingested." | On conflict update every canonical field derived from upstream (`category`, `geom`, `geom_type`, `latitude`, `longitude`, `event_date`, `source_url`, `raw_payload`, `ingested_at`); ensure enrichment trigger reruns when `geom` changes. | patched-local | Add integration test where same `source_id` moves country/state/category/source URL; verify list/detail/map fields update and enrichment is recalculated. |
| SEC-022 | P1 | Medium | Ingestion availability / third-party data abuse | EONET response bodies and raw event payloads are read/stored without size limits. | `api/internal/ingestor/eonet.go` uses `io.ReadAll(resp.Body)`; `normalizer.Normalize` stores full `rawPayload`; `events.raw_payload` is JSONB with no app-level size cap. | Malicious, compromised, or accidentally huge upstream response can spike API memory during ingestion and bloat Postgres with oversized raw payloads. Large payloads can then amplify cache memory usage for public responses. | Wrap reads with `io.LimitReader`, reject oversized responses/events, cap title/source URL/raw payload lengths, and emit a bounded failure alert. | patched-local | Unit test oversized EONET response fails gracefully; integration test confirms oversized raw payload is rejected and ingestion run records a bounded error. |
| SEC-023 | P1 | Medium | Alerting / replay and fatigue | Staleness alert duplicate suppression is process-local only. | `api/internal/alert/watchdog.go` stores `lastAlertReference` inside the watchdog goroutine. | Restart loops, deploy churn, or multiple API replicas can replay the same stale-reference alert repeatedly, exhausting Resend quota and training operators to ignore real incidents. | Persist staleness alert dedupe key in DB, or guard watchdog with DB advisory lock plus durable `last_alert_reference`; add alert rate limits/backoff. | patched-local | Restart API while stale condition remains; confirm only one alert is sent for the same reference time across restart and across two replicas. |
| SEC-024 | P1 | Medium | Ingestion timing / retry abuse | Upstream `retry_after` can stall ingestion for an unbounded duration. | `api/internal/ingestor/eonet.go` sets `sleepSeconds = retry_after + 5` with no upper cap. | EONET or an intermediary returns `429` with a very large `retry_after`; scheduler goroutine sleeps for hours/days, no new countries ingest in that cycle, and watchdog only alerts after damage has started. | Cap retry-after/backoff to a small maximum, enforce max elapsed retry budget, and fail the run when the upstream asks for an excessive delay. | patched-local | Unit test `retry_after=86400` returns bounded failure quickly; watchdog still fires if successful ingestion becomes stale. |
| SEC-025 | P1 | Medium | Frontend / diagnostic amplification | Public frontend renders raw backend ingestion errors to every visitor. | `web/src/components/EventsDashboard.tsx` builds `NASA EONET Ingestion Error: ${apiError}` from `/health`; `/health` currently exposes raw `LastIngestion.Error`. | A transient backend failure can publish provider URLs, internal IPs, proxy behavior, and operational timing on the public landing page. This chains with SEC-003 and helps reconnaissance without touching logs. | Replace public UI text with a generic freshness/degraded message; keep raw error only in private logs/admin diagnostics. | patched-local | Force a failed ingestion in staging; public frontend shows generic degraded copy while operator logs/diagnostics retain the raw error. |
| SEC-026 | P1 | Medium | Scheduler / alert attribution race | Failure alert body can use the global latest ingestion run instead of the current country run. | `api/internal/ingestor/scheduler.go` calls `repo.GetLastIngestionRun(ctx)` after a country failure rather than loading the run ID returned by `Ingest` or querying by country/run. | Under overlapping schedulers or future parallel country ingestion, one country's failure can send an alert containing another country's latest failed run. Operators may chase the wrong country/provider failure. | Return run metadata from `Ingest` or query by current run ID/country; combine with SEC-015 single-runner locking. | patched-local | Concurrent scheduler test with NG/GH failures verifies each alert uses the matching country/run ID. |
